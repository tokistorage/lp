ESSAY — 技術・設計

CloudflareとGAS——便利な依存、見えない代償

無料で高性能。だから使う。
しかし「誰のインフラか」を問わないまま、依存は深くなる。

この記事で言いたいこと：CloudflareとGoogle Apps Script（GAS）は、個人開発者やスタートアップにとって極めて魅力的なインフラだ。無料で高機能、設定も容易。しかしその便利さの裏側には、見えない依存がある。DNSもSSLもトラフィックも他社が握り、ビジネスロジックも他社のランタイムで動く。トキストレージがこれらを使わないのは、技術的な理由ではない。1000年の永続性を目指すとき、すべての外部依存はカウントダウンタイマーになるからだ。

1. 「無料」という誘引

Cloudflareの無料プランは驚異的だ。CDN、DDoS防御、SSL証明書、DNSホスティング、Web Application Firewall。これだけの機能が無料で手に入る。個人サイトから中規模のサービスまで、多くのプロジェクトがCloudflareの無料プランで運用されている。

Google Apps Script（GAS）も同様だ。Googleアカウントがあれば、サーバーレス環境でJavaScriptを実行できる。スプレッドシートとの連携、メール送信、定期実行。バックエンドのないプロジェクトに「ちょっとしたサーバー機能」を追加するのに、これ以上手軽な方法はない。

どちらも優れたサービスだ。技術的な批判をする意図はない。

しかし問いたいのは、「無料で使える」ことと「自分のインフラである」ことは、まったく別の概念だということだ。

無料で使えるインフラは、あなたのインフラではない。
あなたは利用者であり、所有者ではない。

2. Cloudflare——透明な中間者

Cloudflareの本質は「リバースプロキシ」だ。ユーザーのリクエストは、まずCloudflareのサーバーを経由し、そこからオリジンサーバーに転送される。

これはつまり、あなたのサイトへのすべてのトラフィックが、Cloudflareを通過するということだ。SSL/TLSの終端もCloudflareが行う。ユーザーが見ている「安全な接続」は、ユーザーとCloudflare間の暗号化であって、ユーザーとあなたのサーバー間の暗号化ではない。

Cloudflareを導入すると、以下のものが第三者の管理下に入る。

  • DNS——ドメインの名前解決をCloudflareが管理する
  • SSL証明書——暗号化の鍵をCloudflareが発行・管理する
  • トラフィックルーティング——どのリクエストをどこに転送するかをCloudflareが決める
  • キャッシュ——どのコンテンツをどれだけキャッシュするかをCloudflareが制御する
  • アクセス制御——Botと人間の判別、国別ブロックなどをCloudflareが実行する

2022年6月、Cloudflareの大規模障害が発生した。DiscordやShopifyを含む数百万のサイトが同時にダウンした。オリジンサーバーは正常に動いていた。中間者が倒れたとき、その先にあるすべてが倒れたのだ。

Cloudflareが悪いのではない。中間者に依存するアーキテクチャの構造的リスクが、可視化された瞬間だった。

3. GAS——便利な檻

Google Apps Scriptは、Googleのエコシステム内で動作するサーバーレス環境だ。スプレッドシートをデータベース代わりに使い、トリガーで定期処理を実行し、GmailやDriveと連携する。

問題は、すべてがGoogleアカウントに紐づくことだ。

  • 実行環境——GoogleのV8ランタイムで動作する。バージョンアップはGoogleが決める
  • クォータ——1日のメール送信数、スクリプト実行時間、API呼び出し回数、すべてにGoogleが上限を設ける
  • 廃止リスク——GoogleはサービスをEOL（End of Life）にすることで知られている。Google Reader、Google+、Hangouts。GASが永続する保証はない
  • アカウント凍結——Googleアカウントが凍結されれば、GASもスプレッドシートもメールも、すべてが同時に止まる

GASで業務を自動化しているスタートアップは多い。注文処理、在庫管理、顧客連絡。しかしその自動化のすべてが、Googleという一社のプラットフォーム上のゲストとして動いている。

GASは「自分のサーバー」ではない。
Googleの部屋を借りて、Googleのルールで動かしているだけだ。

4. 依存の不可視化

CloudflareもGASも、その依存を感じさせない設計になっている。それが優れたサービスたる所以だ。しかし感じないことと、存在しないことは違う。

制御の対象 GitHub Pages（直接） Cloudflare経由 GAS
DNS 自身で管理可能 Cloudflareが管理 —
SSL/TLS GitHub/Let's Encrypt Cloudflareが終端 Googleが管理
コード実行 クライアントサイド Workers使用時はCloudflare Googleのランタイム
データ保管 リポジトリ内 キャッシュはCloudflare Googleドライブ/スプレッドシート
障害時の影響 GitHub障害時のみ GitHub + Cloudflare障害 Google障害時に全停止
サービス終了リスク Gitで移行可能 DNS移行が必要 コード・データ移行が必要

依存を増やすたびに、障害点（Single Point of Failure）が増える。そして外部依存のSPFは、自分では修復できない。待つしかない。

5. トキストレージの選択

トキストレージはGitHub Pages上の静的サイトだ。Cloudflareは使っていない。GASも使っていない。

Cloudflareを使わない理由

GitHub PagesにはSSLが標準で付与される。CDNもGitHubのインフラで提供される。DDoS防御はGitHubが行う。Cloudflareを追加することで得られるメリットに対して、DNSとSSLの制御を第三者に渡すコストが見合わない。

トキストレージのQRコードは、データをURLパラメータとしてQRコード内に完結させている。サーバーへのリクエストは静的ファイルの配信だけだ。高度なキャッシュ制御もWAFも、構造的に不要なのだ。

GASを使わない理由

トキストレージにサーバーサイドの処理はない。注文管理はBASE、メールはBASEの通知機能、ニュースレターは手作業。自動化の対象となるサーバー処理が、そもそも存在しない。

これは怠慢ではない。サーバー処理を必要としないアーキテクチャを、最初から選んでいるのだ。GASが不要なのは、GASが必要になる設計を避けたからだ。

6. 永続性のための独立

1000年の保存を目指すとき、すべての外部依存はリスクになる。

Cloudflareは2010年設立。GASは2009年公開。どちらも10年台のサービスだ。素晴らしいサービスだが、100年続く保証はない。1000年はなおさらだ。

GitHub Pagesにも同じリスクはある。しかしGitという分散バージョン管理システムに基づいているため、リポジトリをクローンすれば、どこでも再構築できる。GitHubがなくなっても、Gitは残る。コードは手元にある。

Cloudflare経由でDNSを管理していた場合、Cloudflareがなくなればドメインの名前解決が途絶える。GASで業務ロジックを動かしていた場合、GASがEOLになればロジックを書き直す必要がある。

依存の数は、カウントダウンタイマーの数だ。どれかひとつのタイマーが切れたとき、何かが壊れる。タイマーが少ないほど、永続性は高くなる。

便利さには半減期がある。
独立には、半減期がない。

7. 結び——便利さの先を見る

便利なインフラは、借りたインフラだ。

借りたものは、返す日が来る。
返す日が来ないインフラだけが、永続する。

Cloudflareは素晴らしいサービスだ。GASも優れた自動化基盤だ。多くのプロジェクトにとって、最適な選択肢であることは疑いない。

しかしトキストレージが目指しているのは、「多くのプロジェクト」の最適解ではない。1000年後にもデータが読めるという、極端に長い時間軸での設計だ。

その時間軸で見ると、便利さよりも独立が重要になる。高機能よりも、依存の少なさが重要になる。今日のベストプラクティスよりも、100年後に移行可能であることが重要になる。

すべての依存は、いつか終わるサービスへの賭けだ。賭ける対象を最小にすることが、トキストレージの設計原則だ。
