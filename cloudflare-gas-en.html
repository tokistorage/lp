Cloudflare and GAS — Convenient Dependencies, Invisible Costs

Free and powerful. So you use them.
But without asking "whose infrastructure is this," the dependency deepens.

The point of this essay: Cloudflare and Google Apps Script (GAS) are extraordinarily attractive infrastructure for solo developers and startups. Free, feature-rich, easy to set up. But behind that convenience lies invisible dependency. DNS, SSL, traffic routing — all controlled by a third party. Business logic running on someone else's runtime. TokiStorage uses neither, not for technical reasons, but because when you aim for a thousand years of permanence, every external dependency becomes a countdown timer.

1. The Lure of "Free"

Cloudflare's free plan is remarkable. CDN, DDoS protection, SSL certificates, DNS hosting, Web Application Firewall. All of this at no cost. From personal sites to mid-sized services, countless projects run on Cloudflare's free tier.

Google Apps Script (GAS) is similarly compelling. With a Google account, you get a serverless environment for running JavaScript. Spreadsheet integration, email automation, scheduled execution. For projects without a backend, there is no easier way to add "a bit of server functionality."

Both are excellent services. This essay is not a technical criticism of either.

But what must be asked is this: "available for free" and "your own infrastructure" are entirely different concepts.

Infrastructure you use for free is not your infrastructure.
You are a user, not an owner.

2. Cloudflare — The Transparent Middleman

Cloudflare is, at its core, a reverse proxy. User requests first pass through Cloudflare's servers before being forwarded to the origin server.

This means all traffic to your site passes through Cloudflare. SSL/TLS termination happens at Cloudflare. The "secure connection" the user sees is encryption between the user and Cloudflare — not between the user and your server.

When you adopt Cloudflare, the following come under a third party's control:

  • DNS — Cloudflare manages your domain's name resolution
  • SSL certificates — Cloudflare issues and manages your encryption keys
  • Traffic routing — Cloudflare decides which requests go where
  • Caching — Cloudflare controls what is cached and for how long
  • Access control — Cloudflare determines bot vs. human, country-level blocking

In June 2022, a major Cloudflare outage occurred. Millions of sites — including Discord and Shopify — went down simultaneously. Their origin servers were running fine. When the middleman fell, everything behind it fell too.

Cloudflare was not at fault in any moral sense. It was a moment when the structural risk of depending on a middleman became visible.

3. GAS — The Convenient Cage

Google Apps Script is a serverless environment that operates within Google's ecosystem. Use a spreadsheet as a database, run scheduled tasks via triggers, integrate with Gmail and Drive.

The problem is that everything is tied to a Google account.

  • Runtime — runs on Google's V8 engine. Version upgrades are Google's decision
  • Quotas — daily email sends, script execution time, API calls — Google sets all limits
  • Deprecation risk — Google is known for sunsetting services. Google Reader, Google+, Hangouts. There is no guarantee GAS will exist forever
  • Account suspension — if your Google account is frozen, GAS, spreadsheets, and email all stop simultaneously

Many startups automate their operations with GAS. Order processing, inventory management, customer communications. But all of that automation runs as a guest on Google's platform, by Google's rules.

GAS is not "your server."
You are renting a room in Google's building and operating under Google's rules.

4. The Invisibility of Dependency

Both Cloudflare and GAS are designed so you barely feel the dependency. That is precisely what makes them excellent services. But not feeling a dependency and not having one are different things.

Control point GitHub Pages (direct) Via Cloudflare GAS
DNS Self-managed Managed by Cloudflare —
SSL/TLS GitHub / Let's Encrypt Terminated by Cloudflare Managed by Google
Code execution Client-side Cloudflare (if using Workers) Google's runtime
Data storage In the repository Cache at Cloudflare Google Drive / Sheets
Impact of outage Only GitHub outages GitHub + Cloudflare outages All stops on Google outage
Service sunset risk Migrate via Git clone DNS migration required Code and data migration required

Every dependency added is another Single Point of Failure. And external SPFs are ones you cannot fix yourself. You can only wait.

5. TokiStorage's Choice

TokiStorage is a static site on GitHub Pages. It does not use Cloudflare. It does not use GAS.

Why not Cloudflare

GitHub Pages provides SSL by default. CDN is delivered through GitHub's own infrastructure. DDoS protection is handled by GitHub. Against these built-in capabilities, the cost of handing DNS and SSL control to a third party does not justify the marginal benefit.

TokiStorage's QR codes embed data as URL parameters, self-contained within the QR code. Server requests are limited to static file delivery. Advanced cache control and WAF are structurally unnecessary.

Why not GAS

TokiStorage has no server-side processing. Order management runs on BASE. Email uses BASE's notification system. The newsletter is crafted by hand. There is no server-side process to automate in the first place.

This is not neglect. An architecture that requires no server-side processing was chosen from the start. GAS is unnecessary because the design avoids any need for GAS.

6. Independence for Permanence

When aiming for a thousand years of preservation, every external dependency becomes a risk.

Cloudflare was founded in 2010. GAS was released in 2009. Both are services with a history measured in the teens of years. They are excellent services, but none can guarantee a hundred years of operation. A thousand years is out of the question.

GitHub Pages carries the same risk. However, because it is built on Git — a distributed version control system — cloning the repository means the site can be rebuilt anywhere. Even if GitHub disappears, Git remains. The code is in your hands.

If DNS was managed through Cloudflare, name resolution dies with Cloudflare. If business logic ran on GAS, an EOL means rewriting everything.

The number of dependencies is the number of countdown timers. When any one timer expires, something breaks. Fewer timers mean greater permanence.

Convenience has a half-life.
Independence does not.

7. Conclusion — Looking Beyond Convenience

Convenient infrastructure is borrowed infrastructure.

What is borrowed must someday be returned.
Only infrastructure that never needs returning can endure.

Cloudflare is an excellent service. GAS is a capable automation platform. For many projects, they are undoubtedly the optimal choice.

But TokiStorage is not optimizing for "many projects." It is designing for data that can still be read a thousand years from now — an extraordinarily long time horizon.

At that time scale, independence outweighs convenience. Minimal dependency outweighs feature richness. The ability to migrate a hundred years hence outweighs today's best practices.

Every dependency is a bet on a service that will someday end. Minimizing what you bet on is TokiStorage's design principle.
